## Color Contract (60/30/10)

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#1A1A1A` (darkGray) | Section background, overall form surface |
| Secondary (30%) | `#2A2A2A` (midGray) + `#1F1F1F` (cardGray) | Input fields, textarea, confirmation card |
| Accent (10%) | `#B91C1C` (red) | Reserved for specific elements — see list below |
| Destructive | `#B91C1C` (red) | Error borders, validation error text |

**Accent reserved for:**
- Input field focus border ring (all inputs: name, relation, email, phone, textarea)
- The primary CTA button background ("SUBMIT YOUR STORY")
- The 4px left border on the confirmation card
- The animated RedBar above the section heading
- The overline "SHARE YOUR STORY" text color
- Inline validation error text and error border on required fields

Accent is NOT used on: placeholder text, inactive input borders, body copy, labels, the permission disclaimer.

**Source:** Inherited from Phase 2 UI-SPEC. Extended to cover email, phone, and textarea focus states.

## Spacing Scale

Multiples of 4 only. Identical to Phase 2 — no new tokens.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, label-to-error gap |
| sm | 8px | Gap between stacked inputs on mobile, gap between file rows, gap between input and validation error |
| md | 16px | Default element spacing, input padding (horizontal), gap between form columns, gap between form sections |
| lg | 24px | Section horizontal padding |
| xl | 32px | Confirmation card horizontal padding, top margin on submit button area |
| 2xl | 48px | Drop zone vertical padding, section top/bottom internal padding, confirmation card vertical padding |
| 3xl | 64px | Not used within the form component — reserved for page-level |

**Exceptions:**
- All input fields use `padding: 12px 16px` (vertical 12px x2 + 20px line height = 44px, meets 44px touch target minimum).
- Textarea uses `padding: 12px 16px`, `minHeight: 120px` (3 visible lines approximate).
- The form grid collapses to single column on mobile (< 640px viewport) — gap remains 16px between stacked rows.

**Source:** Inherited from Phase 2 UI-SPEC + `app/components/ShareStory.tsx` measured inline styles.

## Component Inventory

Phase 3 extends `app/components/ShareStory.tsx` without extracting new component files (consistent with Phase 2 pattern).

### ShareStory (existing — extend in place)
- **File:** `app/components/ShareStory.tsx`
- **Current state:** Has name + relation inputs, upload zone, file list, progress bar, error banner, submit button, confirmation screen. Submit currently triggers only on all-files-done.
- **Phase 3 deliverables:** Add email + phone inputs, textarea for text story, client-side validation, Airtable metadata POST via `/api/submit/metadata`, text-only submission path to confirmation, and error state for metadata POST failure.

### New inputs to add (all within ShareStory, no new files)

**Email input (FORM-03 — optional)**
- Type: `<input type="email">`
- Placeholder: "Email address (optional)"
- Padding: `12px 16px`
- Width: full column in a 2-column grid row alongside phone
- Validation: no required validation; format validation only if non-empty (must contain @)
- Error: "Please enter a valid email address." — appears below field, Source Sans 3 14px weight 600, color `#B91C1C`

**Phone input (FORM-04 — optional)**
- Type: `<input type="tel">`
- Placeholder: "Phone number (optional)"
- Padding: `12px 16px`
- Width: full column in a 2-column grid row alongside email
- Validation: no required validation; no format validation in v1

**Textarea for text story (FORM-05 — conditional required)**
- Type: `<textarea>`
- Placeholder: "Type your story here — a memory, what he meant to you, something only you know."
- Padding: `12px 16px`
- Width: full width (spans both columns)
- Min height: 120px (approximately 3 visible lines)
- Resize: vertical only (`resize: vertical`)
- Character limit: none in v1
- Validation: required if no file selected (FORM-06)
- Error: "Please add a file or type your story — we need at least one." — appears above the submit button when both are absent

### Form layout order (top to bottom inside the flex column)
1. Row 1 — 2-column grid: Name (required) | Connection to Coach McCollum (required)
2. Row 2 — 2-column grid: Email (optional) | Phone (optional)
3. Row 3 — full width: Textarea (conditionally required)
4. Row 4 — full width: UploadZone (existing)
5. Row 5 — full width: File list rows (conditional on files present)
6. Row 6 — full width: ProgressBar (existing)
7. Row 7 — full width: ErrorBanner (existing — upload errors)
8. Row 8 — full width: Validation error for FORM-06 (new — shown when submit blocked)
9. Row 9 — left-aligned: Submit button
10. Row 10 — full width: Permission disclaimer (existing)

**Mobile grid collapse:** On viewports narrower than 640px, both 2-column grids collapse to 1 column (stack vertically). Gap remains 16px between stacked fields.

### Confirmation screen (existing — extend for text-only path)
- Currently triggers only when `allDone === true` (all files uploaded successfully)
- Phase 3 must also trigger when: no files AND `textStory.trim().length > 0` AND metadata POST succeeds
- The confirmation copy does not change
- The name interpolation pattern `Thank you{name ? ', ${name}' : ''}.` does not change

### New API route: `/api/submit/metadata`
- Called after file upload completes (or instead of upload for text-only submissions)
- POST body: `{ name, relation, email, phone, textStory, fileKey, timestamp }`
- On success: `setSubmitted(true)`
- On failure: show MetadataErrorBanner (see below)

### MetadataErrorBanner (new inline element — not a new component file)
- Rendered inline in ShareStory, same appearance as existing ErrorBanner
- Trigger: `/api/submit/metadata` POST returns non-2xx or network error
- Copy: "We couldn't save your submission. Your file is safe — please try again." with a "Try again" inline button
- "Try again" re-triggers the metadata POST only (does not re-upload the file)
- Appearance: same as ErrorBanner — `background: #7F1D1D20`, `border: 1px solid #B91C1C`, `padding: 12px 16px`, Source Sans 3 14px weight 600, color `#B91C1C`

## Validation Rules

| Field | Required | Format Rule | Error Trigger |
|-------|----------|-------------|---------------|
| name | Yes | Any non-empty string | On submit attempt |
| relation | Yes | Any non-empty string | On submit attempt |
| email | No | Must contain @ if non-empty | On blur (field-level) + on submit |
| phone | No | None in v1 | Never |
| textStory | Conditional | Non-empty if no files selected (FORM-06) | On submit attempt |
| file | Conditional | Accepted MIME type (UPLD-05 — existing) | Immediately on selection |

**Validation order on submit:** name → relation → email → FORM-06 (file OR text story). First failing check blocks submit and shows its error. Multiple errors are not shown simultaneously — show one at a time in top-to-bottom field order.

**State required to be blocked:** submit button is disabled (opacity 0.4, cursor not-allowed) when:
- Upload is in progress, OR
- Metadata POST is in progress, OR
- No files AND textarea is empty

All other states: button is active (user may attempt submit and receive field-level validation errors if name/relation empty).

## Accessibility

Inherit all Phase 2 accessibility contracts. Additional for Phase 3:

- All new inputs must have `aria-label` or visible `<label>` element. Because placeholders are used instead of floating labels, add `aria-label` matching the placeholder text on each input.
- Textarea must have `aria-label="Your story"`.
- Validation errors must be announced via `role="alert"` on the error container, or by associating with `aria-describedby` on the relevant input.
- The FORM-06 validation error above the submit button must use `role="alert"` so screen readers announce it on submit attempt.
- Required fields (name, relation) must have `aria-required="true"`.
- The MetadataErrorBanner must use `role="alert"` (same as existing ErrorBanner).
- Submit button `disabled` attribute must be set when the button is in disabled state (not just visual opacity) — this ensures screen readers announce it as unavailable.
